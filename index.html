Compatibility nobody typed in.

A knowledge graph for a used PC-parts inventory. Assert that a motherboard fits a CPU, and the graph derives the reverse, which listings offer compatible parts, and which builds have enough power — computed by two reasoning paradigms over one ontology, not stored.

01 — Two reasoning paradigms

One graph, two kinds of inference — kept distinguishable by predicate.

Each edge color in the visualizations tells you which engine derived it. The tools were chosen for fit: OWL where class semantics and property chains shine, SPARQL rules where value-matching does.

compatibleWith — OWL, reasoned by HermiT

Assert one direction of compatibility and the reasoner composes the rest: a symmetric property plus the chain sells ∘ compatibleWith makes offersCompatible listings appear that nobody linked by hand. Materialized offline, served as plain triples.

# asserted (one direction)
tdy:mobo_x1 tdy:compatibleWith tdy:cpu_x1 .

# inferred by HermiT
tdy:cpu_x1 tdy:compatibleWith tdy:mobo_x1 .
tdy:listing_7 tdy:offersCompatible tdy:mobo_x1 .

fitsWith — SPARQL CONSTRUCT rules

Physical and electrical fit, derived from part attributes across three rule patterns — equality (socket, RAM generation, interface), set-containment (form factor, GPU clearance), numeric threshold (PSU wattage) — plus whole-build power aggregation.

# rules/01-socket.rq
CONSTRUCT { ?cpu tdy:fitsWith ?board }
WHERE {
  ?cpu   a tdy:CPU ;         tdy:socket ?s .
  ?board a tdy:Motherboard ; tdy:socket ?s .
}

02 — Pipeline

From CSV to a served, reasoning-backed endpoint in one command.

01CSV intake8 part sheets, validated before anything is generated
02RDF mappingPython + rdflib emit catalog.ttl — no hand-written triples
03Inference7 SPARQL rules + HermiT OWL reasoning, materialized offline
04SHACLShapes validated through two engines off one file
05ServeFuseki + TDB2 in Docker — docker compose up
06QuerySPARQL endpoint, local-LLM natural language, these visualizations

03 — What it demonstrates

Each tool chosen for fit, not name.

  • OWL 2 modeling — a defined class and a property chain, classified offline with HermiT
  • SPARQL CONSTRUCT rules — three derivation patterns plus whole-build power aggregation
  • Two inference paradigms in one served graph, distinguishable by predicate
  • Tested CSV intake pipeline — validate, then map; replaces hand-written RDF
  • SHACL validation — one shapes file, two engines
  • Provider-agnostic local-LLM layer — English to SPARQL against the live endpoint
  • Deployable artifact — persistent triplestore, containerized, one command
